¿Qué son los requerimientos de Software?

Es el proceso por el cual, mediante el uso de herramientas y técnicas obtendremos una descripción de características y comportamientos que un sistema debe tener, basándonos en la especificación del cliente o por las necesidades de los usuarios.

Características de un requerimiento

• Conciso: que sea fácil de leer y entender para su respectiva interpretación.

• Ambiguo: el requerimiento debe estar sujeto a una única interpretación, no puede haber términos que tenga más de una acepción.

• Consistente: los requerimientos no deben ser contradictorios con otros.

• Completo: todas las características y comportamientos deben estar definidas.

• Verificar o comprobar: antes de ejecutar el software se debe hacer una inspección para verificar que si cumpla con todos los requisitos establecidos.

• Especificado por escrito: “Las palabras vuelan, lo escrito se queda” (Cayo Tito), se debe hacer un contrato por escrito.

Técnicas y herramientas para identificar los requerimientos

El objetivo es obtener información de un grupo de personas para poder desarrollar requerimientos más exactos.

• Lluvia de ideas: también llamada tormenta de ideas, es una actividad grupal la cual facilita el surgimiento de ideas sobre un problema en un ambiente creativo.

• Prototipo: se crea un modelo a través de la captura de requerimientos y estos se definen con el cliente.

• Casos de usos: que es describir los posibles escenarios que puedan ocurrir por la interacción de los usuarios con el sistema.

En el desarrollo de Software existen dos tipos de requerimientos.

Requerimientos funcionales: es la parte en la que se define que funciones o comportamientos debe tener la aplicación o el sistema de software.

Ejemplos:

• El usuario pueda registrarse a la pág.

• El usuario pueda loguearse.

• Pueda hacer compras.

• Pueda agregar más de un producto a la canasta.

• Pueda eliminar productos de la canasta.

• Pueda ver la lista de pedidos.

Requerimientos No funcionales: consiste en definir todas las cualidades y restricciones que un producto debe tener en el proceso de desarrollo.

Ejemplos:

• Requerimientos de usabilidad: que vendría siendo el cómo, el usuario entiende el funcionamiento del software, para su ingreso de datos e interpretación de los resultados de esta.

• Requerimientos de eficiencia: es la medición de rendimiento del software relacionado con el número de operaciones por segundo, tiempo de respuesta, así como el consumo de recursos (memoria, procesador y espacio en red).

• Requerimientos de dependibilidad: consiste en una serie de atributos las cuales son:

• Disponibilidad: que es operabilidad del sistema relacionado con el servicio que presta.

• Confiabilidad: Que es que el sistema cumpla con las funciones requeridas en condiciones establecidas.

• Integridad: que es que el sistema no presente alteraciones inadecuadas.

• Mantenibilidad: que sea posible modificar o repara algún proceso sin afectar la continuidad del sistema.

Sabiendo que funcionalidades debe tener la página web o aplicación, pasaremos a un conjunto de actividades a la cual se le denomina Ingeniería de requerimientos que tiene como tarea el generar especificaciones precisas sin ambigüedades, compactas y claras sobre el comportamiento del sistema.

Estas actividades serian:

Obtención de requerimientos: este proceso será la base de la construcción de todo el proyecto, el cual los analistas del sistema deberán trabajar junto al cliente en una junta llamada kick off (Arranque) para especificar qué problema debe resolver el sistema o aplicación.

Por lo general se debe especificar lo siguiente:

• Objetivo, fecha de inicio y fecha final.

• Presentación del equipo.

• Presentación del stakeholder.

• Fecha de reunión con el cliente.

Análisis de requerimiento: en esta etapa lo que se busca es resaltar los problemas o encontrar errores por medio del análisis de los requerimientos obtenidos, para encontrarles soluciones o alternativas mediante la investigación o el intercambio de ideas con el equipo.

Actividades de la etapa de análisis.

Reducir ambigüedades del requerimiento: que el requerimiento empleado no sea mal interpretado, para esto se debe eliminar términos que tengan más de una acepción.

Traducir a lenguaje técnico los requerimientos: los requerimientos con menos ambigüedades serán tratados para llevarlos al lenguaje técnico.

Plantear un modelo lógico: se utiliza diagramas de flujo o cualquier otra representación para el modelado. Consiste en crear un modelo que cumpla con los requerimientos y luego discutir los detalles con el cliente.

Especificaciones de requerimientos: en esta parte se documenta los requerimientos detalladamente con el cliente. Se documenta a detalle las características, funcionalidades, el alcance del sistema y la forma en que hará sus funciones. Se define los requerimientos funcionales y no funcionales.

Verificación de Requerimientos: se debe seleccionar diferentes revisores de diferentes disciplinas como analistas, ingenieros de software y arquitectos los cuales deben tener conocimiento de los estándares de documentación de la organización, para lo cual se prepara un checklist que podría tener los siguientes puntos.

• ¿Están incluidas todas las funcionalidades requeridas por el cliente?

• ¿Existen conflicto entre los requerimientos?

• ¿Algunos de los requerimientos tiene más de una interpretación?

• ¿Están todos los requerimientos claramente representados?

• ¿Lo requerimientos pueden ser implementados con la tecnología y presupuesto disponible?

• ¿Las especificaciones están escritas en el lenguaje apropiado?

• ¿Es flexible?

• ¿Está claramente definido el origen de cada requerimiento?

• ¿Es escalable?

• ¿Los requerimientos pueden ser expuestos a pruebas unitarias?

Aceptación de requerimientos: es cuando el analista se reúne con el cliente para comenzar a leer el documento donde se encuentra las especificaciones de todos los requerimientos ayudándose incluso de prototipos, con la intención de que el cliente le quede claro el funcionamiento del sistema. Una vez el cliente acepte se deberá realizar un compromiso formal ya sea por correo o por medio de un documento que contenga la firma del cliente.

Administración de requerimientos: Es una actividad que se realiza durante todo el proyecto y consiste en llevar un control de todos los cambios, así como que cada etapa cumpla con las fechas especificadas para no causar retrasos con la fecha de entrega estipulada.

Importancia de los requerimientos:

1. Determina el alcance del proyecto

2. Direcciona al equipo de trabajo

3. Reconoce posibles riesgos

4. Generar buena comunicación

5. Evita reprocesos
